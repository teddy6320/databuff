# 测试指南

## 当前现状

当前仓库的 [package.json](/package.json) 没有提供 `test` 脚本，仓库里也没有成体系的 `tests/` 或 `*.spec.*` 用例目录。因此目前更贴近真实现状的做法是：

- 先保证构建通过
- 再做针对变更范围的人工回归

## 基础检查

代码改动完成后，建议至少执行：

```bash
yarn build
```

这个命令会先执行 TypeScript 检查，再进行 Vite 生产构建，是当前仓库里最可靠的通用校验方式。

## 人工回归建议

### 页面类改动

- 页面能否正常进入
- URL query 是否按预期回写
- 时间范围切换是否正常
- 列表、图表、抽屉、弹窗是否还能打开
- 详情页下钻和返回链路是否正常

### API 类改动

- 请求方法是否正确
- 查询参数或请求体是否正确传递
- 成功态、空态、报错态是否都能被页面正确处理
- 是否影响到已有环境标签、管理域或登录失效处理

### Store / Utils 类改动

- 依赖它们的核心页面是否还能正常初始化
- 是否引入了空值、缓存失效或重复请求问题
- 是否和现有时间范围、授权状态、面包屑逻辑兼容

## 建议的回归入口

如果你修改的是通用能力，优先选这些高频页面做烟雾验证：

- [智能告警模块](/docs/modules/alarm-center/README.md)
- [应用性能监控模块](/docs/modules/app-monitor/README.md)
- [基础设施模块](/docs/modules/infrastructure/README.md)
- [系统管理模块](/docs/modules/sys-manage/README.md)

这些模块覆盖了列表、图表、详情、配置、权限和跨模块跳转，比较容易暴露回归问题。

## 文档类改动

如果只是修改 Markdown 文档：

- 检查链接是否可达
- 检查路径、接口名、路由参数是否和代码一致
- 检查 `docs/README.md` 与 `docs/DOC-STRUCTURE.md` 是否同步

## 后续可扩展方向

虽然当前仓库还没有正式测试体系，但如果后续要补自动化，建议从下面三类开始：

1. `utils` 下的纯函数
2. `api` 层的参数转换逻辑
3. 交互稳定、输入输出清晰的共享组件

这样落地成本最低，也最容易先建立稳定收益。
