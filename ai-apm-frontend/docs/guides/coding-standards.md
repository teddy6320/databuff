# 编码规范

## 目标

这份规范不是重新定义一套理想风格，而是总结当前仓库里已经稳定存在的写法，尽量减少“新代码和旧代码不像一套”的问题。

## 总体原则

- 优先保持同目录、同模块的既有风格
- 业务页面负责交互编排，不直接承担底层请求封装
- 能复用已有组件、Store、工具函数时，不重复造轮子
- 文档、路由、API 三者如果发生变化，尽量一起更新

## TypeScript 与 Vue 约定

从 [tsconfig.json](/tsconfig.json) 和现有源码来看，项目当前约定是：

- Vue `2.7`
- TypeScript `strict: true`
- 支持装饰器语法
- SFC 以 `.vue` 为主，业务逻辑辅助文件放在同目录 `.ts`

## 代码风格

当前仓库里更常见的写法是：

- 使用单引号
- 语句末尾保留分号
- 使用 2 空格缩进
- import 路径优先使用 `@/` 别名引用 `src`

这不是通过 ESLint 强制的统一格式，所以新增代码时最重要的是“跟随邻近文件”。

## 命名建议

| 场景 | 建议 |
|------|------|
| 页面目录 | 跟随现有模块命名，通常使用 kebab-case 或已有业务缩写 |
| 页面入口文件 | 优先使用 `index.vue` |
| 组件文件夹 | 复用现有命名风格，不强行统一大小写 |
| API 方法 | 使用 `get/create/update/delete/save` 等动作前缀 |
| Store mutation | 使用全大写常量或显式字符串常量 |

## 页面开发约定

- 路由页面通常落在 `src/views/<module>/.../index.vue`
- 列表页的筛选、图表、表格、弹窗通常继续拆在当前目录
- 路由参数和时间 query 需要和 [src/router/index.ts](/src/router/index.ts) 的处理逻辑保持兼容
- 如果页面依赖全局时间范围，优先复用 Store 里的时间状态，而不是自己维护一套

## 请求与状态约定

- 页面层不要直接 import `axios`
- 统一通过 [src/utils/axios.ts](/src/utils/axios.ts) + `src/api/*` 发请求
- 公共缓存、权限、时间、服务树等全局数据优先放在 `store`
- 如果只是单页面瞬时状态，优先保留在页面内部，不要过早提升到 Vuex

## 组件使用约定

- 高频基础组件优先复用 [docs/components/README.md](/docs/components/README.md) 已记录的组件
- 只有跨模块、高复用的组件才考虑放进 `src/components`
- 强业务耦合组件优先放在模块自己的 `views/<module>/` 目录里

## 注释与文档

- 复杂图表装配、接口兼容、历史路径兼容逻辑可以写短注释
- 不要给简单赋值和显然逻辑添加噪音注释
- 影响入口、参数、接口的改动，建议同步更新对应文档

## 提交前自检

- 路径别名是否正确
- 是否复用了现有 API / Store / 组件
- 路由 query 是否与既有页面兼容
- 业务接口是否放进了 `src/api/*`
- 至少本地执行过一次 `yarn build`
